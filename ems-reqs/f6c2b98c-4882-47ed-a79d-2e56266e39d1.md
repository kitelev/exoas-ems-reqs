# req(ems): utility commands hidden on Done — «Create Related Task» shown only on Done

## Job Story

When I finish a task, I want its buttons to drop the create / convert / prioritize
/ plan / maintenance actions that only make sense on active work, and instead
offer «Create Related Task» to spin off a follow-up — so a completed task reads
as a closed record with the one forward action that fits, not a cluttered
active-work toolbar.

## Statement (Gherkin)

```gherkin
Scenario: A utility command is hidden once the task is Done
  Given an ems__Task in status Done (not archived)
  And a utility command among {Convert to Project, Create Action, Clean Properties,
      Copy Label to Aliases, Repair Folder, Fix Missing Label, Rename to UID}
      whose precondition previously evaluated true on that Done task
  When the command button-set for the task is resolved (binding-match ∩ precondition-eval)
  Then that utility command is not visible

Scenario: The same utility command is still shown on an active task
  Given the same utility command
  And an ems__Task in status Backlog or Doing satisfying that command's data/guard condition
  When the command button-set is resolved
  Then that utility command is visible (behavior-preserving on active tasks)

Scenario: «Create Related Task» is shown only on a Done task
  Given the «Create Related Task» command bound to ems__Task
  When the target task is in status Done and not archived
  Then «Create Related Task» is visible
  And when the target task is in status Backlog or Doing «Create Related Task» is hidden
  And when the target task is archived «Create Related Task» is hidden

Scenario: «Create Action» and «Create Related Task» carry opposite status gates
  Given «Create Action» and «Create Related Task» previously shared one precondition
  When each is repointed to its own AllPrecondition wrapper
  Then «Create Action» is visible on Backlog/Doing and hidden on Done
  And «Create Related Task» is hidden on Backlog/Doing and visible on Done
```

## Verification

**Binding class:** Integration — a production-shape test builds a real triple
store from vault fixtures, resolves the button-set for an ems__Task through the
real `CommandResolver` + `PreconditionEvaluator`, and asserts visibility across
statuses. Revert-verify: removing the «Visible until Done» conjunct from a gated
command's wrapper re-exposes it on a Done task (RED); restoring it hides it
(GREEN). For «Create Related Task»: reverting its Done-only wrapper to the
not-Done/shared precondition flips its visibility (visible on Backlog/Doing RED)
and restoring returns it to Done-only (GREEN).

**Ground-truth scope (verified via `resolve-buttons`, the authoritative
button-visibility oracle, before writing this requirement):** on real
non-archived Done ems__Task assets the current button-set leaks {Convert to
Project, Create Action, Create Related Task} plus the data-gated {Clean
Properties, Copy Label to Aliases} (and, on a data-broken Done task, {Repair
Folder, Fix Missing Label, Rename to UID}). **Seven** commands get the new
not-Done gate — Convert to Project, Create Action, Clean Properties, Copy Label to
Aliases, Repair Folder, Fix Missing Label, Rename to UID. **One** command —
Create Related Task — gets the inverse Done-only gate and is split from the
precondition it shared with Create Action. The five planning/voting commands named
in the target matrix (Vote on Effort, Plan for Evening, Plan on Today, Shift Day
Backward, Shift Day Forward) are **already** gated to not-Done by the shared
composite `c57642c4` («Visible until Done, not a prototype») and are verified
hidden on Done — so this requirement adds no gate for them (its observable
end-state is already achieved for that family).

**Design note (homoiconic, zero engine code):** each not-Done gate is a
per-command `AllPrecondition` wrapper nesting the command's existing (archived-gate)
precondition composite AND the reusable «Visible until Done» atomic (37517554).
«Create Related Task» is repointed to a wrapper `AllPrecondition[«Allow re-open
(Done + not-prototype)» 2bee97b7, «Not archived» 28f722ba]`. No shared atomic and
no already-shipped requirement asset (Partially Done / Un-archive / Archived-gate /
Create Next Iteration) is mutated. No plugin/CLI engine code changes.

**Cross-class note:** because a command's precondition is class-agnostic (the data
model has no per-binding precondition), the not-Done gate on commands that also
bind to other effort classes (Convert to Project → ems__Project; the exo__Asset
maintenance commands) also hides them on a Done asset of those classes. This is
semantically consistent (a completed effort offers no re-planning/maintenance) and
unavoidable in the current model.
