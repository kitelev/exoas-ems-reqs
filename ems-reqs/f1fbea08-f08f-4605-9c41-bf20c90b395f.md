# req(ems): archived tasks show only «Un-archive» — all other task commands hidden

## Job Story

When I open a task I have archived, I want to see only the one action that makes
sense on an archived thing — «Un-archive» — and none of the lifecycle, create,
convert, prioritize, plan, or maintenance buttons, so an archived task reads as a
frozen record with a single obvious way back into active work.

## Statement (Gherkin)

```gherkin
Scenario: An archived task shows only «Un-archive»
  Given an ems__Task in status Done carrying exo__Asset_archived "true"
  When the command button-set for the task is resolved (binding-match ∩ precondition-eval)
  Then the only visible command is «Un-archive»
  And no other task-bound command is visible

Scenario: The archived-guard hides a task command on an archived task
  Given a task-bound command whose precondition previously evaluated true on a Done task
  And an archived-guard conjunct (NOT EXISTS { $target exo:Asset_archived "true" }) added to that command's precondition
  When the target task is archived
  Then that command is hidden
  And when the target task is not archived that command is still shown as before

Scenario: «Un-archive» itself is not archived-gated
  Given the «Un-archive» command (visible only when the task is archived)
  When the target task is archived
  Then «Un-archive» remains visible
```

## Verification

**Binding class:** Integration — a production-shape test builds a real triple
store from vault fixtures, resolves the button-set for an archived ems__Task
through the real `CommandResolver` + `PreconditionEvaluator`, and asserts the
resolved set equals exactly `{Un-archive}`. Revert-verify: removing the
archived-guard from one gated command's precondition re-exposes that command on
the archived task (RED); restoring it returns the set to `{Un-archive}` (GREEN).

**Ground-truth scope (verified via `resolve-buttons`, the authoritative
button-visibility oracle, before writing this requirement):** on a real archived
ems__Task the current button-set is `{Un-archive, Create Action, Re-open, Convert
to Project, Create Related Task, Clean Properties, Copy Label to Aliases}` — six
commands leak beyond «Un-archive». Three further data-gated maintenance commands
(Repair Folder, Fix Missing Label, Rename to UID) leak on a data-broken archived
task. These **nine** commands get the archived-guard. The remaining task commands
named in the target matrix (Start Effort, Mark Done, Rollback to Backlog/Draft,
Move to Backlog, Set Draft Status, Trash, Vote on Effort, Set Criticality
High/Medium/Low) are **already hidden on a Done task** by their existing
status-gate, and since an archived task is a Done task they are already hidden on
archived — so no additional gate is added for them (the requirement's observable
end-state is achieved without redundant edits).

**Design note (homoiconic, zero engine code):** the archived-guard is a single
reusable `AtomicPrecondition` (`ASK { FILTER NOT EXISTS { $target
exo:Asset_archived "true" } }`), added as one conjunct of a per-command
`AllPrecondition` wrapper. The wrapper (not in-place ASK editing) is required
because two gated preconditions are `hostFunction` atomics (`hasObsoleteProperties`,
`isInWrongFolder`) that carry no `sparqlAsk` text, and because `847c37e0`
(«Target is not a prototype») is shared by six commands and must not be mutated in
place. No plugin/CLI engine code changes.

**Cross-class note:** because a command's precondition is class-agnostic (the
data model has no per-binding precondition), gating a command that also binds to
other classes (Re-open, Create Action → ems__Project; the maintenance commands →
exo__Asset) also hides that command on an archived asset of those other classes.
This is semantically consistent (a frozen archive offers no lifecycle/maintenance)
and unavoidable in the current model; «Un-archive» itself remains bound to
ems__Task only.
