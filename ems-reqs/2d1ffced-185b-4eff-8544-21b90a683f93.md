# req(ems): «Create Next Iteration» re-bound on Done task, spawns Backlog copy with all properties

## Job Story

When I have **finished** a task that recurs — a cycle I will run again next time
— I want one action on the completed task that queues a fresh copy of it in my
backlog, carrying all of its context forward and linked back to the iteration it
followed, so the next cycle starts from where this one left off without me
re-typing anything. This is the honest-success counterpart to «Partially Done»
(the failure path): «Create Next Iteration» never touches the finished task, it
only spawns the next one.

## Statement (Gherkin)

```gherkin
Scenario: Create Next Iteration spawns a Backlog copy from a Done task
  Given an ems__Task in status Done
    and the task is not a prototype
  When I invoke the «Create Next Iteration» command on the task
  Then a new ems__Task is created in status Backlog
    And the new task copies every substantive property of the original task
      (label, body, area, parent, zone, relates, deadline, time-estimate)
    And the new task links to the original via ems__Effort_prevIteration
    And the new task does not carry ems__Effort_partiallyDone
    And the new task does not carry the original's start / end / resolution timestamps
    And the new task has a fresh exo__Asset_uid and exo__Asset_createdAt
    And the new task is co-located with the original (same folder / isDefinedBy)
    And the original Done task is left unchanged

Scenario: Create Next Iteration is only offered on a Done task
  Given the «Create Next Iteration» command is bound to ems__Task
  When the target task's ems__Effort_status is Backlog or Doing
  Then the «Create Next Iteration» command is not shown
  And when the target task is archived (Done + archived) the command is not shown
  And when the target is a prototype the command is not shown
```
