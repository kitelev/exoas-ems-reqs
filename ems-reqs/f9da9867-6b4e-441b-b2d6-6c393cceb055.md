# req(ems): «Un-archive» restores an archived task from the archive

## Job Story

When a task I had archived turns out to still matter — I picked it back up, or
archived it by mistake — I want one action on the archived task that lifts it out
of the archive, so it reappears in my active views without my having to hand-edit
frontmatter.

## Statement (Gherkin)

```gherkin
Scenario: Un-archive removes the archive flag from an archived task
  Given an ems__Task carrying exo__Asset_archived "true"
  When I invoke the «Un-archive» command on the task
  Then the task's archived flag is removed
  And the task is once again visible in active views

Scenario: Un-archive is only offered on an archived task
  Given the «Un-archive» command is bound to ems__Task
  When the target task carries exo__Asset_archived "true"
  Then the «Un-archive» command is shown
  And when the target task is not archived the «Un-archive» command is not shown
```

## Verification

**Binding class:** Integration — a production-shape test builds a real triple
store, resolves and executes the «Un-archive» command through the real
`CommandResolver` + `GroundingExecutor`, and asserts the archived flag is gone
from the resulting file state; it also asserts the command's precondition ASK is
true for an archived target and false for a non-archived one.

**Design note (homoiconic, zero engine code):** the plugin `ArchiveAssetService`
archives in place — it sets the top-level `archived: "true"` frontmatter flag (and
strips `aliases`) with **no physical file move**. Un-archive is therefore the pure
inverse: a `property_delete` grounding removing the `archived` frontmatter key.
No new grounding service is needed (`unarchiveAsset` remains a separate cross-vault
CLI concern, not this command).
