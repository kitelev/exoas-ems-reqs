# req(ems): Trashed is a second terminal status equal to Done

## Job Story

When I trash a task, I want it to behave as a genuinely finished thing — just
like a Done task: I can archive it away with one button, and none of the
planning / voting / create / convert / maintenance buttons that only make sense
on active work are shown — so a trashed task reads as a closed record with an
archive path, not a discarded item that still offers a full active-work toolbar
yet can't even be archived.

## Statement (Gherkin)

```gherkin
Scenario: A Trashed task can be archived
  Given an ems__Task in status Trashed that is not archived
  When the command button-set for the task is resolved (binding-match ∩ precondition-eval)
  Then «Archive Completed» is visible
  And when the same task is Done and not archived «Archive Completed» is still visible (behavior-preserving)

Scenario: Active-work commands are hidden on a Trashed task
  Given an ems__Task in status Trashed
  And an active-work command among {Execute, Plan on Today, Plan for Evening,
      Shift Day Backward, Shift Day Forward, Vote on Effort, Create Action,
      Convert to Project, Clean Properties, Copy Label to Aliases, Repair Folder,
      Fix Missing Label, Rename to UID} whose precondition previously evaluated
      true on that Trashed task
  When the command button-set is resolved
  Then that active-work command is not visible
  And the same command is still hidden on a Done task (Trashed treated exactly like Done)

Scenario: Active-work commands are still shown on an active task
  Given the same active-work command
  And an ems__Task in status Backlog or Doing satisfying that command's data/guard condition
  When the command button-set is resolved
  Then that active-work command is visible (behavior-preserving on active tasks)

Scenario: A Trashed + archived task shows only «Un-archive»
  Given an ems__Task in status Trashed carrying exo__Asset_archived "true"
  When the command button-set is resolved
  Then the only visible command is «Un-archive»
  And no active-work command is visible

Scenario: «Is terminal» drives both the archive gate and the active-work gate
  Given a reusable «Is terminal status (Done or Trashed)» precondition
  When it is AND-ed with «Not archived» it forms the «Archive Completed» gate (positive direction)
  And when it is negated («Visible until terminal») it replaces «Visible until Done» in the active-work gates (negative direction)
  Then a change to the definition of «terminal» propagates consistently to both directions

Scenario: «Mark Done Post Factum» stays available on a Trashed task
  Given «Mark Done Post Factum» keeps its «not-Done» gate (the shared atomic 37517554 is not mutated)
  When the target task is Trashed
  Then «Mark Done Post Factum» remains available (a mistakenly-trashed task can be corrected Trashed→Done)
```

## Verification

**Binding class:** Integration — a production-shape test builds a real triple
store from vault fixtures, resolves the button-set for an ems__Task through the
real `CommandResolver` + `PreconditionEvaluator`, and asserts visibility across
statuses (Backlog/Doing/Done/Trashed × archived). Revert-verify
([[integration-test-revert-verify]]): reverting the «Is terminal» conjunct back
to «Done»-only re-exposes an active-work command on a Trashed task and hides
«Archive Completed» on a Trashed task (RED); restoring returns the terminal
semantics (GREEN).

**Ground-truth scope (verified via `resolve-buttons`, the authoritative
button-visibility oracle, before writing this requirement):**
- On a real **Trashed + not-archived** ems__Task the current button-set is
  `{Create Action, Convert to Project, Plan on Today, Plan for Evening, Shift Day
  Backward, Shift Day Forward, Vote on Effort, Clean Properties}` (8 visible) —
  «Archive Completed» is **absent** (the archive bug) and eight active-work
  commands **leak**.
- On a real **Trashed + archived** ems__Task the current button-set is
  `{Un-archive, Plan on Today, Plan for Evening, Shift Day Backward, Shift Day
  Forward, Vote on Effort}` (6 visible) — five planning commands leak past
  «Un-archive».
- The active-work family gated only «not-Done» is **thirteen** commands: the six
  sharing `c57642c4` (Execute + five planning) and the seven utility commands
  from `f6c2b98c` (Create Action, Convert to Project, Clean Properties, Copy
  Label to Aliases, Repair Folder, Fix Missing Label, Rename to UID). All
  thirteen get the terminal gate.

**Design note (homoiconic, zero engine code):** a single reusable «Is terminal
status (Done or Trashed)» `AtomicPrecondition` is the source of truth for
terminality, used in both directions — positively (`AllPrecondition[Is-terminal,
Not-archived 28f722ba]` becomes the «Archive Completed» gate, replacing «Is Done
and not archived» 8762ddc2) and negatively (its `NotPrecondition` «Visible until
terminal» replaces the «Visible until Done» conjunct 37517554 in the active-work
gates). The shared composite `c57642c4` is edited in place (swap
37517554→«Visible until terminal», AND «Not archived» 28f722ba) because all six
of its consumers want the identical extended semantics. The seven utility
commands are gated by the **wrap-the-wrapper** pattern: a new per-command
`AllPrecondition[<utility-status-gate composite>, «Visible until terminal»]`
nests the already-shipped utility-gate composite one level deeper and the command
is repointed to it — the merged utility-gate composites (`f6c2b98c`) are **not
mutated** (2-level composite nesting is supported by
`CommandResolver.loadPreconditionSubject` recursion, depth-cap 10). The shared
«Visible until Done» atomic (37517554) is **not** mutated — «Mark Done Post
Factum» keeps its not-Done gate so a mistakenly-trashed task can be corrected
Trashed→Done. No shipped requirement asset (Partially Done / Un-archive /
Archived-gate / Create Next Iteration / Utility-status-gates) is mutated. No
plugin/CLI engine code changes.

**Cross-class note:** because a command's precondition is class-agnostic (the
data model has no per-binding precondition), the terminal gate on commands that
also bind to other effort/asset classes (Convert to Project → ems__Project; the
exo__Asset maintenance commands) also hides them on a Trashed asset of those
classes, and «Archive Completed» becomes available on a Trashed ems__Project /
ems__Meeting as well. This is semantically consistent (a terminal effort of any
class offers no active work and is archivable) and unavoidable in the current
model. «Un-archive» itself is unchanged.

**No data migration:** the ~57 already-Trashed+archived tasks in the live vault
are already in their terminal end-state; this requirement broadens the
precondition (forward), it does not migrate data (backward).
