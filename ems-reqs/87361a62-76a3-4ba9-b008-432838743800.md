# req(ems): «Partially Done» closes a Doing task partially and spawns a next iteration

## Job Story

When I am working a task and realise I cannot finish it in this pass — but the
work still matters and deserves a second attempt — I want one action that marks
the current task as done-but-partial and immediately queues a fresh copy of it in
my backlog, so the closed task records that the finish fell short while the
follow-up work is preserved as a linked next iteration.

## Statement (Gherkin)

```gherkin
Scenario: Partially Done closes the Doing task and spawns a next iteration
  Given an ems__Task in status Doing
    and the task is not a prototype
  When I invoke the «Partially Done» command on the task
  Then the original task's ems__Effort_status becomes Done
    And the original task gets an ems__Effort_endTimestamp and ems__Effort_resolutionTimestamp
    And the original task's ems__Effort_partiallyDone is set to "true"
    And a new ems__Task is created in status Backlog
    And the new task copies every property of the original task
    And the new task links to the original via ems__Effort_prevIteration
    And the new task does not carry ems__Effort_partiallyDone
    And the new task does not carry the original's start / end / resolution timestamps
    And the new task has a fresh exo__Asset_uid and exo__Asset_createdAt

Scenario: Partially Done is only offered on a Doing task
  Given the «Partially Done» command is bound to ems__Task
  When the target task's ems__Effort_status is Backlog, Done, or Archived
  Then the «Partially Done» command is not shown
  And when the target is a prototype the command is not shown
```
